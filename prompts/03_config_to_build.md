# Промпт-инструкция 3: Конфигурация → Сборка

> Финальный промпт для Claude Code, с которого начинается автономная реализация MVP. Предполагает, что пакет конфигурации уже разложен в репозитории (CLAUDE.md, SPEC.md, .claude/*). Стек — зафиксирован.

---

## Промпт (копируй в чат Claude Code в корне репозитория с уже разложенной конфигурацией)

```
Ты — Claude Code в корне репозитория CyberBook. В нём уже есть:
- CLAUDE.md (главный конфиг)
- SPEC.md (техническая спецификация)
- SPEC_TEMPLATE.md (шаблон фичи)
- PROJECT_IDEA.md (контекст продукта)
- .claude/agents/ (5 субагентов)
- .claude/rules/ (5 правил)
- .claude/skills/ (2 skill'а)

Контекст:
- Стек зафиксирован (см. CLAUDE.md «Стек»). Если там всё ещё TBD — ОСТАНОВИСЬ и попроси пользователя зафиксировать стек одним сообщением перед сборкой.
- Этот промпт запускает Фазу 2 «Полный MVP» из PROJECT_IDEA.md («8. План запуска»).

Задача:
Реализовать MVP согласно приоритезации в SPEC.md («Приоритет / Зависимости» каждого модуля + итоговая таблица). Это модули 1–11.

Workflow:

ЭТАП 1 — Инициализация репозитория
1. Прочитай CLAUDE.md и все файлы в .claude/ — они регулируют весь стиль работы.
2. Прочитай SPEC.md целиком, зафиксируй модели данных.
3. Инициализируй проект по стеку (package.json, tsconfig, linters, etc.).
4. Настрой env-менеджмент (.env.example) с перечнем переменных по стеку.
5. Настрой PWA-каркас (manifest.json, service worker).

ЭТАП 2 — Фундамент БД (делегировать database-architect)
6. Одной большой миграцией или серией миграций создай все сквозные таблицы MVP:
   users, clubs, club_members, zones, stations, club_maps, bookings, booking_stations, user_reputation, notifications_log, otp_codes, audit_log
7. Для каждой — RLS, индексы, ON DELETE, CHECK constraints.
8. SECURITY DEFINER функции: apply_reputation_event, create_booking (атомарная), auto_no_show, auto_complete, set_updated_at trigger.

ЭТАП 3 — Серверная логика (делегировать backend-engineer)
9. Реализуй все API-эндпоинты MVP из SPEC.md модули 1–11.
10. Cron-задачи: auto_no_show, auto_complete, process_notifications, send_reminders.
11. Real-time канал для админки: подписка на bookings по club_id.
12. Очередь уведомлений (notifications_log + worker).
13. SMS-интеграция для OTP (провайдер — на усмотрение, рекомендуемый: Mobizon для KZ).

ЭТАП 4 — Frontend (делегировать frontend-developer)
14. Публичная витрина: /c/:slug с картой, зоны, слоты, Bottom-sheet бронирования, OTP inline, QR после успеха.
15. /login, /signup/owner, /invite/:token.
16. /admin — дашборд; /admin/today-board — полноэкранный; /admin/bookings, /admin/stations, /admin/zones, /admin/map-editor, /admin/settings.
17. /my/bookings — мои брони для игроков.
18. PWA manifest, service worker, offline-fallback для витрины.
19. i18n: RU default, KZ и EN как добавочные locales.

ЭТАП 5 — Тесты
20. Unit-тесты бизнес-логики: overlap-проверка, расчёт цены, пересчёт репутации.
21. Integration-тесты RLS: user A не видит bookings user B; admin клуба X не видит bookings клуба Y.
22. E2E для критического пути: регистрация → бронь → QR → check-in админом → completed.

ЭТАП 6 — Review (делегировать qa-reviewer)
23. Запусти qa-reviewer на всю реализованную работу.
24. Разбери отчёт: все BLOCKER и MAJOR должны быть закрыты.

ЭТАП 7 — Готовность
25. Обнови CLAUDE.md разделом «Команды разработчика» (конкретные dev/build/test команды).
26. Создай README.md с 3 секциями: как запустить локально, как применить миграции, как задеплоить.
27. Напиши 1-2 seed-скрипта: тестовый клуб, тестовый админ, 40 ПК в 3 зонах, 5 тестовых броней.

ПРАВИЛА выполнения:

- Активно делегируй субагентам. Сам выполняй только координацию.
- Следуй .claude/rules/*: RLS обязательна, SQL-типы стандарт, i18n, четыре состояния UI, идемпотентность платежей.
- Спрашивай пользователя ТОЛЬКО если:
  a) Стек не зафиксирован в CLAUDE.md
  b) Неясны внешние интеграции (какой SMS-провайдер? какая платёжка для MVP?)
  c) Возник конфликт между SPEC.md и PROJECT_IDEA.md — эскалируй, не выбирай сам.
- Используй Context7 MCP перед написанием любого кода, работающего с внешним API или библиотекой — свежая документация.
- Прогресс показывай через TodoWrite. По одному пункту in_progress за раз.

НЕ делай:
- ❌ Не пиши кода до применённой миграции.
- ❌ Не реализуй v2-модули (blacklist, promotions, payments, subscriptions, analytics, super-admin) без явного запроса.
- ❌ Не меняй структуру `.claude/` без обсуждения.
- ❌ Не «упрощай» RLS-политики, если не уверен в безопасности.
- ❌ Не откладывай тесты на «потом».

Итог:
Работающий MVP, который можно запустить локально одной командой, применить миграции второй, посеять тестовые данные третьей, и пройти сквозной путь «ссылка в bio → бронь → check-in» четвёртой.
```

## Когда использовать

- В день старта сборки MVP (после фиксации стека и разложенной конфигурации)
- При крупной перекатке (снос старого кода, свежий старт по новой спеке)

## Критерии успеха после сборки

- `supabase migration up` (или аналог) применяет все миграции без ошибок
- `pnpm dev` (или аналог стека) поднимает приложение
- На `/c/<slug-тестового-клуба>` виден интерактивный план зала
- Можно пройти путь: выбрать ПК → ввести телефон → OTP → бронь создана → видна в `/my/bookings`
- Админ логинится → видит бронь в `/admin/today-board` → нажимает check-in → статус меняется
- `qa-reviewer` не выдаёт `BLOCKER`
