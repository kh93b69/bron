# Промпт-инструкция 1: Идея → Спецификация

> Используется, если нужно перегенерировать или расширить `SPEC.md` из `PROJECT_IDEA.md`. В базовой поставке `SPEC.md` уже готов — этот промпт нужен для апдейтов (например, добавили новый раздел в идею → обновили спеку).

---

## Промпт (копируй в чат Claude с загруженным PROJECT_IDEA.md и SPEC.md)

```
Ты — старший продуктовый архитектор, специализирующийся на SaaS для B2B-ниш.
Твоя задача — сгенерировать/актуализировать техническую спецификацию CyberBook (SaaS для бронирования ПК в кибер-клубах) на основе PROJECT_IDEA.md.

Контекст:
- Загружен PROJECT_IDEA.md (документ идеи)
- Загружен (возможно, устаревший) SPEC.md
- Технический стек: TBD (используй SQL-стандартные типы и нейтральные термины)

Задача:
Произвести полную спецификацию, покрывающую ВСЕ модули из PROJECT_IDEA.md (раздел «5. Архитектура → Модули»).

Для КАЖДОГО модуля обязательно описать 6 блоков:
1. User Stories — «Как [роль], я хочу [действие], чтобы [результат]» (минимум 3, включая edge-кейсы)
2. Модель данных — таблицы с типами (uuid, text, jsonb, timestamptz, integer, boolean), FK с ON DELETE, RLS-политики (SELECT/INSERT/UPDATE/DELETE)
3. API — эндпоинты в формате: `METHOD /path`, body, ответ 2xx/4xx с кодами ошибок
4. Экраны и компоненты — путь, ключевые UI-элементы, состояния (loading/empty/error/success)
5. Бизнес-логика — правила валидации, формулы (цена, рейтинг), переходы статусов
6. Крайние случаи — race conditions, недоступность внешних сервисов, граничные значения

Дополнительно в начале SPEC.md:
- Сквозные сущности (все таблицы с полями)
- Индексы
- Глобальные RLS-политики
- Non-functional требования
- Таблица приоритезации MVP vs v2

Правила:
- Конкретика вместо абстракций. «Продвинутая аналитика» → «дашборд с heatmap occupancy × weekday, revenue по дням, top-5 ПК».
- Никаких TODO, [заполнить], «уточнить».
- Типы полей как в SQL. Деньги — integer в тенге. Время — timestamptz.
- API-коды ошибок — строковые константы (BOOKING_SLOT_OCCUPIED, USER_BANNED, RATE_LIMITED).
- Формат документа — markdown с оглавлением в начале.

Workflow:
1. Прочитай PROJECT_IDEA.md целиком.
2. Прочитай существующий SPEC.md (если есть) — определи, что уже покрыто.
3. Составь список модулей из раздела «Модули» идеи.
4. Для каждого модуля сгенерируй 6 блоков.
5. Собери сквозные сущности — единую схему БД.
6. Проверь совместимость: все FK существуют, RLS покрывает все таблицы, API-ответы консистентны.
7. Выдай итоговый SPEC.md одним документом.

ГЛАВНОЕ ПРАВИЛО:
Не спрашивай — добавляй сам, если ответ есть в PROJECT_IDEA.md. Уточнение — только если в идее противоречие или совсем нет данных (тогда один блок «Требуются уточнения» в конце, не более 5 пунктов).

Язык вывода — русский (тело), identifiers — английские.
Формат — markdown. Документ сразу готов к коммиту.
```

## Когда использовать

- После расширения `PROJECT_IDEA.md` (добавили новую аудиторию, новый модуль, новый риск)
- После изменения монетизации (новые тарифы → новый раздел в спеке)
- При ревизии спеки раз в квартал

## Критерии качества вывода

- Покрыты все модули из идеи
- Для каждого модуля — 6 блоков из методологии
- Типы полей — SQL-стандарт
- API-эндпоинты в едином формате
- Явно выделены MVP vs v2
