# Skill: implement-feature

## Вход

- Готовая спецификация фичи в формате `SPEC_TEMPLATE.md` (в PR-описании или в `features/YYYY-MM-DD_<slug>.md`)
- Опционально: issue/ticket c дополнительным контекстом

## Алгоритм

### Шаг 1 — Подтверди спеку
1. Прочитай спецификацию фичи.
2. Прочитай соответствующий раздел `SPEC.md` (системная спека модуля).
3. Если в спеке фичи есть противоречие с `SPEC.md` — остановись, запроси уточнение.
4. Если спека содержит `TODO`, `[заполнить]`, «уточнить у PM» — остановись, нужно доработать спеку.

### Шаг 2 — План выполнения (TodoWrite)
Разбей на подзадачи:
- [ ] Миграция БД (если нужна)
- [ ] Бизнес-логика (server functions)
- [ ] API-эндпоинты
- [ ] UI-компоненты
- [ ] i18n ключи
- [ ] Тесты (unit + RLS + e2e критического пути)
- [ ] Review через `qa-reviewer`

### Шаг 3 — БД первым (если нужна миграция)
Делегируй `database-architect`:
- Создание миграции (используй skill `create-migration`)
- RLS-политики
- Индексы
- `SECURITY DEFINER` функции (если требуется)
- Применить локально и проверить `psql` командами

**Блокер**: не идти дальше, пока миграция не применена на локальной БД и не прошла smoke-проверка RLS.

### Шаг 4 — Серверная логика (backend-engineer)
Параллельно с UI (если response-схема зафиксирована):
- Валидация входа (zod)
- Проверки бизнес-правил
- Транзакции + FOR UPDATE где нужно
- Уведомления через `notifications_log`
- Audit log
- Rate-limiting

### Шаг 5 — UI (frontend-developer)
- Роутинг / новые страницы
- Компоненты (shadcn/ui first)
- Все 4 состояния (loading/empty/error/success)
- Формы с zod-валидацией
- i18n-ключи (ru + kk + en)
- Optimistic UI где уместно
- Real-time подписки для админки

### Шаг 6 — Если фича касается платежей — payments-specialist
Не делегировать backend-engineer:
- Эндпоинты создания платежа / депозита
- Webhook-обработчики
- Идемпотентность
- Возвраты
- Гейтинг фич через `hasFeature()`

### Шаг 7 — Тесты
- Unit для чистой бизнес-логики (pricing, overlap, reputation, feature-gating)
- Integration для RLS (user A не видит данных user B, club A не видит club B)
- E2E для критического флоу фичи
- При фичах с платежами: тест на повтор webhook, тест на неверную подпись

### Шаг 8 — Review (qa-reviewer)
Запусти субагента `qa-reviewer` на готовый diff. Дождись отчёта со структурой BLOCKER / MAJOR / MINOR. Не закрывать фичу при наличии BLOCKER.

### Шаг 9 — Финальная проверка
- [ ] `CLAUDE.md` не изменён случайно
- [ ] `SPEC.md` обновлён, если поведение существующего модуля меняется
- [ ] Миграция применима на staging одной командой
- [ ] В `.env.example` добавлены новые env (если есть)
- [ ] CHANGELOG / release notes обновлены (если ведутся)

## Когда субагентов запускать параллельно

- `database-architect` + `frontend-developer` (UI-скелет на моках) — параллельно
- `backend-engineer` + `frontend-developer` (если API-контракт зафиксирован) — параллельно
- `qa-reviewer` — всегда ПОСЛЕ остальных

## Когда строго последовательно

- БД → Backend (нельзя писать server-код до готовой схемы)
- Backend → Payments (платёж опирается на `bookings`)
- Everything → qa-reviewer

## Антипаттерны (не делай этого)

- ❌ Начинать UI без зафиксированной API-схемы
- ❌ Коммитить без ручного прогона хотя бы happy-path в браузере
- ❌ Откладывать тесты на «потом»
- ❌ Пропускать `qa-reviewer` «потому что фича маленькая»
- ❌ Пропускать обновление `SPEC.md` при изменении поведения модуля
